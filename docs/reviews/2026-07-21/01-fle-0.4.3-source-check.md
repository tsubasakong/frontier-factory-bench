## 1. 自定义 mod 挂载 —— **部分成立**(机制可行,但无接线的公开扩展点)

**证据**

- 容器启动**总是**挂载一个 mods 目录并传 `--mod-directory /opt/factorio/mods`,Factorio 会按其中 `mod-list.json` 加载 mod——这是标准机制,FLE 不拦截:
  - `fle/cluster/run-envs.sh` 生成的 compose:`image: factoriotools/factorio:2.0.73`,volumes 含 `- source: ./mods target: /opt/factorio/mods`。`fle/cluster/mods/` 里**只有一个 `mod-list.json`,仅启用 `base`**。
  - `fle/cluster/run_envs.py:243`(`ComposeGenerator.services_dict`)挂的是 `_bundled_mods_volume()`——指向 **pip 包内** 的 `fle/cluster/mods`(经 `importlib.resources`)。
- 看似预留的钩子都是死代码:
  - `run-envs.sh` 的 `-m/--attach_mods` 构造了 `MODS_VOLUME` 变量,但 compose heredoc 里从未引用它;
  - `run_envs.py:135` 的 `_mod_path()`(支持 `FLE_MODS_PATH` 环境变量)和 `_mods_volume()` 定义了,但 `services_dict` 里注释明写 `# Note: attach_mod overlays user mods on top of bundled mods (not currently used)`(`run_envs.py:247`)。
- 可用的绕行路径(均不需 fork):
  - `ComposeGenerator` 是公开类,构造参数直接接受 `pkg_scenarios_dir` / `pkg_config_dir`(可指向外部目录);子类化覆盖 `_bundled_mods_volume()` 指向你自己的 mods 目录(自定义 mod + 启用 base 的 mod-list.json)即可。compose 文件生成在 `FLE_STATE_DIR/docker-compose.yml`,也可以生成后改写或干脆自带 compose。
  - scenario 名在 `run_envs.py` 不校验(shell 脚本硬校验只允许 `open_world|default_lab_scenario`);`save_file` 参数支持挂 `.zip` 存档(校验含 `level.dat`)。
  - Inspect sandbox 镜像(`fle/eval/inspect/sandbox/Dockerfile`)是另一条路径:`FLE_MODS_DIR`/`FLE_TOOLS_DIR` 环境变量可覆盖 FLE 的 Lua 库和工具目录(`fle/env/utils/rcon.py:36-49`)——注意这是 FLE 的 RCON Lua 脚本目录,**不是** Factorio mod 目录。

**影响**:不需要 patch FLE Python 运行时,但"官方支持的 mod 注入参数"不存在;适配层要自己生成/改写 compose(或子类化 `ComposeGenerator`)。这条建议写进 ADR,并对 `run_envs.py` 的未接线钩子保持警惕——上游将来接线时行为可能变。

## 2. 新原型枚举 —— **部分成立**(Python 侧确有硬编码枚举过滤,Lua 侧是通用的)

**证据**

- `fle/env/game_types.py:83` `class Prototype(enum.Enum)`:约 200 个硬编码成员,值是 `(名称, 实体类)` 元组;同文件还有硬编码的 `RecipeName`、`ResourceName`、`Technology`、`prototype_by_name` 查找表。
- **强制 Prototype 枚举的工具**(自定义内容直接被拒):`place_entity`(`entity.value` 解构)、`get_entity`(`assert isinstance(entity, Prototype)`)、`craft_item`、`insert_item`(assert)、`extract_item`、`can_place_entity`、`set_entity_recipe`(非 Prototype/RecipeName 直接 `raise ValueError`)。
- **观测过滤**:`fle/env/tools/agent/get_entities/client.py:118` 对 Lua 返回的每个实体在 `Prototype` 里找匹配,找不到就打印 `Warning: No matching Prototype found` 然后 `continue`——**自定义实体被静默丢弃**。
- **字符串友好的工具**:`get_prototype_recipe`(`Union[Prototype, RecipeName, str]`)、`set_research`/`get_research_progress`(`hasattr(technology, "value")` 回退)、`inspect_inventory`(`Inventory` 是 `extra="allow"` 的 pydantic,自定义物品可见)、`harvest_resource`(按位置)。Lua server 端完全按字符串名通用处理(如 `get_entities/server.lua` 的 `find_entities_filtered{name=...}`、`get_production_stats/server.lua` 的 `force.get_item_production_statistics`)。
- 任务验证侧不受影响:`ThroughputTaskConfig.throughput_entity: Union[str, Prototype]`,`calculate_achievements` 按物品名字符串统计——自定义 `compute-unit` 可直接做吞吐验证目标。

**影响**:默认 action/observation 层里自定义机器"放不下、看不见"。绕行方案有两条:(a) 运行时向 `Prototype` 枚举注入成员(改 `_member_map_`/`_value2member_map_`,可行但脆弱);(b) 更干净——用 `FLE_TOOLS_DIR` 指向"FLE tools 副本 + 自有工具"目录,`lua_manager.setup_tools` 会遍历该目录注册任意 `client.py`/`server.lua` 工具对,写一套接受字符串的 `place_entity`/`get_entities` 变体。Python Enum 不可子类扩展,这是硬约束,adapter 必须正面解决它。

## 3. Gym 接口状态 —— **部分成立**(可用且有测试,但不在 CI、有未文档化的坑)

**证据**

- 0.4.3 中 `fle/env/gym_env/`(environment/registry/action/observation + README)存在,`tests/gym_env/` 有 5 个测试文件;但 CI(`.github/workflows/factorio-test.yml:45`)只跑 `tests/actions/`,**gym_env 测试不进 CI**。
- **issue #197 已于 2025-07-09 关闭(completed)**(gh api 实测)。0.4.3 的 gym_env 已按 REPL 模型重写:Action = `{agent_idx, code, game_state}`,就是"提交一段 Python 代码"。你引用的"与 REPL 脱节"是 2025-05 提 issue 时的状态,0.4.3 上这个具体担忧已不成立;真正的现状是"已对齐 REPL,但维护强度弱于核心 action 层"。
- 依赖是**旧 `gym` 包**(pyproject 里 `"gym"`,非 gymnasium),`environment.py` 顶部还有 numpy 2.x 兼容 hack(`np.bool8 = np.dtype(np.bool)`)。
- 注册机制:`registry.py:178` 模块导入即 `register_all_environments()` → 遍历内置 task_registry → `gym.register(id=task_key, entry_point="...registry:make_factorio_env", kwargs={"spec": spec})`。env id 就是裸 task_key(如 `steel_plate_throughput`;README 里的 `Factorio-...-v0` 格式已过时)。
- **坑 1**:`registry.py:96` `def make_factorio_env(spec, run_idx)` 的 `run_idx` **无默认值**——`gym.make("id")` 会直接 TypeError,必须 `gym.make("id", run_idx=0)`(FLE 自己在 `fle/env/protocols/_mcp/state.py:51` 和 `fle/eval/inspect/integration/solver.py:240` 都这么调;而 `example_usage.py` 里 `gym.make(example_env_id)` 的写法实际会报错)。
- **坑 2**:`TaskFactory.create_task` 只认内置 task_registry 的 key(`task_factory.py` → `task_registry.create_task`,未注册即 KeyError)。README 宣称的 `_registry.register_environment(...)` 自定义注册路径在 task 创建环节走不通。
- **外部注册可行**:`FactorioGymEnv(instance, task)` 构造是公开的,`TaskABC` 可子类化(tests/gym_env 就是直接这么构造的)。你的仓库可以 `gym.register(id="FrontierFactory-AIInfra-v0", entry_point="your_pkg:make_env")`,在 entry point 里自建 `FactorioInstance` + 自定义 Task + `FactorioGymEnv`——完全绕过封闭的 registry/factory。

**影响**:`gym.make("FrontierFactory-AIInfra-v0")` 的目标成立,但实现路径是"自注册 + 直接构造 `FactorioGymEnv`",而不是 FLE 的 registry 扩展点。注意 `gym` 与 `gymnasium` 混用问题(FLE 的 spaces 对象是 `gym.spaces`),建议在 adapter 里统一。

## 4. 任务验证模型 —— **成立**

**证据**

- `FactorioGymEnv.step()` 每步调用 `self.task.verify(...)`(`environment.py`),terminated = verify 结果。
- `fle/eval/tasks/throughput_task.py` 的 `ThroughputTask.verify` 正是你要的模式:循环执行 `eval_program_with_achievements(program=f"sleep({self.holdout_wait_period})")` —— **holdout 窗口内 agent 零动作**,工厂自由运转,之后用产量统计差分(`calculate_achievements` 的 dynamic 项)算吞吐;`success = max_achieved_throughput >= quota`。`holdout_wait_period`/`pre_holdout_wait_period` 可配。
- no-op 存在:`sleep` 工具(`fle/env/tools/agent/sleep/`)。server.lua 只做 `storage.elapsed_ticks += seconds*60`;client 按 game speed 换算成墙钟时间 sleep,期间服务器自由运行。
- 时间模型要清楚:`storage.elapsed_ticks` 是**虚拟动作成本计数器**(sleep/craft/harvest/move 累加,见 `craft_item/server.lua:168`、`move_to/server.lua:171` 等),**不是 `game.tick`**;真实的工厂产量靠墙钟 × speed 流逝产生。

**影响**:spec 的"untouched verification window + 持续产量"可以直接复用 `ThroughputTask` 的模式甚至子类;自定义物品名(string)可直接作 `throughput_entity`。

## 5. 观测模型 —— **成立**(两个细节要注意)

**证据**

- `reset()` 返回 `(observation.to_dict(), info)`;`step()` 返回 5 元组。`Observation`(`fle/env/gym_env/observation.py`)字段:`raw_text`、`map_image`(base64 PNG)、`entities`、`inventory`、`research`、`game_info`、`score`、`automated_score`、`flows`、`task_verification`、`messages`、`serialized_functions`、`task_info`、`character_positions`。
- 生产统计可经 action 层获取:`namespace._get_production_stats()`(admin 工具,在命名空间里以下划线名暴露,REPL 里其实也能调)返回 force 级 item/fluid 产量统计,Lua 端通用、**包含 mod 物品**;observation 的 `flows` 即来源于它。research 状态(`save_research_state/server.lua`)通用遍历 `force.technologies`,自定义科技可见。
- 细节 1:`entities` 观测经过 `get_entities`,受第 2 条的枚举过滤——自定义机器默认不进观测。
- 细节 2:`reset(seed=...)` 的 docstring 明写 `seed: Not used`(`environment.py:540`)——**seed 被忽略**,deterministic reset 依赖静态 scenario 地图 + reset 流程,不是 seed。

**影响**:观测结构满足 spec 列举的字段,但"observation 里识别新实体"依赖第 2 条的枚举扩展方案落地;spec 里 `reset(seed=N)` 的用户故事需要自己在 scenario/adapter 层实现,FLE 不提供。

## 6. 确定性 —— **不成立**(指"逐 tick 可重放";任务级结果可做到稳定)

**证据(非确定性来源)**

- **动作落点漂移**:gym step 开始即 `set_speed_and_unpause(10)`(`environment.py`),agent 代码执行期间服务器以 10x 自由运行,每条 RCON 命令落在哪个 tick 取决于墙钟和网络栈;同一动作序列两次执行落点 tick 必然漂移。
- **sleep 是墙钟近似**:`real_world_sleep = ticks_added / 60 / game_speed`(`sleep/client.py`),实际经过的游戏 tick 取决于服务器是否跑满 speed(负载导致 UPS 下降时,holdout 窗口实际产量会变)。
- RCON 存在瞬时 `[processing]` 错误,靠重试吸收(CHANGELOG 0.4.0)。
- seed 链路缺失:gym `reset(seed=)` 忽略 seed;map seed 固定在集群层 `map_gen_seed = 44340`(`run_envs.py:63`,且只有 open_world 才传;`run-envs.sh` 总是传)。
- 正面证据:reset 路径 Lua(`reset`/`regenerate_resources`/`clear_entities`/lab scenario)无 `math.random`,静态图重置后实体级状态是确定的。

**影响**:spec 第 18 条"同 seed 同动作序列 → 相同任务结果 + 关键状态哈希"需要重新表述:逐 tick 世界状态哈希在 FLE 层不成立;可成立的是"任务结果(吞吐阈值判定)+ 容差内的指标"可复现。若要 tick 精确窗口,得在自研 mod 的 control 阶段做(`on_tick` 计数 + `game.tick_paused`,FLE 的 `GameControl.pause()` 已经在用 `tick_paused`),这把"自己的 mod 不只是 data stage"变成了硬需求。

## 总结

**"独立扩展仓库不 fork"在 0.4.3 上站得住**,但适配层工作量比启动方案预期的大,集中在四个点,且每点都有不需 fork 的出路:

1. **mod 注入**:子类化 `ComposeGenerator`(或自带 compose)指向自己的 mods/scenarios 目录。无官方 flag,但也不需要改 FLE 代码。
2. **Prototype 枚举**:这是最硬的一堵墙(`place_entity`/`get_entities` 等核心工具都过它)。首选 `FLE_TOOLS_DIR` 注册字符串版自定义工具;枚举运行时注入作为备选。
3. **环境注册**:绕过封闭的 `FactorioGymRegistry`/`TaskFactory`,自己 `gym.register` + 直接构造 `FactorioGymEnv(instance, TaskABC子类)`。
4. **确定性**:放弃逐 tick 哈希断言,验证用"holdout 窗口 + 吞吐阈值 + 余量";tick 精确性需求下沉到自研 mod 的 control 阶段。

**如果只能做一件事 de-risk**:不要先写美术或经济系统,先做一个 1-2 天的 **垂直 spike**:一个只含 1 个 item + 1 台机器的最小 mod,用子类化 `ComposeGenerator` 起容器,然后实测打通整条最险链路——`place_entity` 放下自定义机器(枚举注入和 `FLE_TOOLS_DIR` 两条路都试)、`get_entities` 能看见它、`sleep(holdout)` 期间它的产量出现在 `_get_production_stats()` 且 `ThroughputTask.verify` 式的判定能通过。第 1、2 条假设是项目根基且只有绕行方案、尚未有人验证过;第 3、4、5、7 条在 0.4.3 源码里已有直接证据支持,风险低。这个 spike 过了,独立仓库决策就可以签字;过不去,再启动 fork 也符合 bootstrap 文档里"证明扩展点缺失才 fork"的既定政策。
